# 2 控制流的第一课：while、if/else 与分支指令

现代嵌入式编程第1课建立了“顺序执行”的基本模型：代码从上到下，PC 逐步前进。现实程序当然不可能永远线性执行，必须根据条件走不同路径，并且重复执行某些片段。

第2课的重点就是控制流（control flow）：

1. 循环：`while`
2. 分支：`if/else`
3. 条件判断背后的位运算习惯写法

这也是后续状态机、调度、实时系统的第一块基石。

## Lesson-02 源码主线

lesson-02 的核心代码非常经典：

```c
int main(void) {
    int counter = 0;
    while (counter < 21) {
        ++counter;
        if ((counter & 1) != 0) {
            /* odd path */
        }
        else {
            /* even path */
        }
    }
    return 0;
}
```

它同时演示了三件事：

1. 有界循环（直到条件不满足退出）。
2. 条件分支（奇偶路径分离）。
3. 用按位与判断奇偶（`counter & 1`）。

## while 循环在机器层的样子

`while (counter < 21)` 并不是“高级概念”，编译后通常是：

1. 比较（`CMP`）
2. 条件跳转（`BGE`/`BLT` 等）
3. 回跳到循环头（`B`）

这里要特别注意：`CMP` 不会把“比较结果数值”写回通用寄存器，它会更新 `APSR`
（Application Program Status Register）中的条件标志位（主要是 `N/Z/C/V`），后续
`BGE`、`BEQ`、`BLT` 等条件跳转就是读取这些标志位来决定是否跳转。

概念汇编可写成：

```asm
LOOP_HEAD:
    LDR   R0, [counter]
    CMP   R0, #21
    BGE   LOOP_END

    ADDS  R0, R0, #1
    STR   R0, [counter]

    ... if/else ...
    B     LOOP_HEAD

LOOP_END:
```

这说明循环的本质就是“比较 + 跳转 + 回边”。

## if/else 的底层结构

`if ((counter & 1) != 0)` 先计算表达式，再分支：

```asm
LDR   R0, [counter]
ANDS  R1, R0, #1
BEQ   EVEN_PATH   ; 若结果为0走偶数分支

ODD_PATH:
    ...
    B   AFTER_IF

EVEN_PATH:
    ...

AFTER_IF:
```

关键点：高级语言中的“真假判断”，在机器层通常是设置条件标志位（flags）后执行条件分支。

在 ARM Cortex-M（如 STM32F103）里，这些 flags 就在 `APSR` 中；例如前面的
`ANDS` 会更新 `Z` 位，所以 `BEQ` 本质上是在判断 `APSR.Z == 1`。

## 为什么用 `counter & 1` 判断奇偶

数学上：一个整数最低位为 1 表示奇数，为 0 表示偶数。

因此：

- `counter & 1 == 1` → odd
- `counter & 1 == 0` → even

相比 `counter % 2`，按位与通常更直接，且更贴近底层思维。在许多编译器上 `%2` 也会被优化成位运算，但显式写位运算有助于教学和可解释性。

## 控制流图（CFG）直觉

把这段程序画成控制流图，会得到：

1. 入口块（初始化 `counter`）
2. 循环判断块（`counter < 21`）
3. 循环体块（自增）
4. if 判断块（奇偶）
5. odd 块 / even 块
6. 回边到循环判断
7. 退出块（`return 0`）

这张图的价值在于：

- 可以直观看到“路径数量”。
- 后面做代码覆盖率和测试时，会用到“每条边是否执行”。

## 调试时应重点观察什么

第 2 课推荐在调试器里观察：

1. `counter` 每轮变化是否符合预期（1 到 21）。
2. odd/even 路径是否交替命中。
3. 退出条件触发时 PC 是否跳到循环后。

如果看到 `counter` 没变化或者跳转异常，通常是：

- 条件写错（如 `<` 与 `<=` 混淆）。
- 自增位置放错。
- 变量被优化器处理后观察窗口不直观（可调低优化级别）。

## 常见控制流错误

### 1. 无限循环

例如把更新语句漏掉：

```c
while (counter < 21) {
    // 忘记 ++counter
}
```

结果条件永远为真。

### 2. 边界 off-by-one

`< 21`、`<= 21`、`< 20` 的行为不同，循环次数分别会差 1。

### 3. 条件表达式副作用

若把复杂副作用塞进 `if` 条件，调试难度会明显上升。初学阶段建议条件表达式保持纯粹。

典型例子：

```c
// 例1：自增混入条件，判断后 i 已经变化
if (i++ < 10) {
    ...
}

// 更清晰的写法
if (i < 10) {
    ...
}
++i;
```

```c
// 例2：把赋值写进条件，容易和 == 混淆
if (flag = read_button()) {
    ...
}

// 更清晰的写法
flag = read_button();
if (flag != 0) {
    ...
}
```

```c
// 例3：读取硬件寄存器并清标志（读操作带副作用）
if (USART1->SR & RXNE_FLAG) {
    data = USART1->DR;  // 读取 DR 可能同时清除 RXNE
    ...
}

// 更稳妥：先把状态和动作拆开，副作用更可见
uint32_t sr = USART1->SR;
if ((sr & RXNE_FLAG) != 0U) {
    data = USART1->DR;
    ...
}
```

共同问题是："判断逻辑" 和 "状态修改" 混在同一行，单步调试时不容易快速定位 bug。


## 代码风格建议（从第二课就开始）

虽然示例很短，依然建议保持工程习惯：

1. 条件表达式写成可读形式（必要时加括号）。
2. `if`/`else` 块都用花括号，避免未来改动引入歧义。
3. 对位运算意图写简短注释（如 odd/even）。

这能显著降低后续维护成本。
